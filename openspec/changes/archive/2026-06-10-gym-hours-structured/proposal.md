# Proposal: Gym Hours Structured Form

## Intent

Replace the free-text `horario: String?` field on the Gym singleton (where admins could write anything, producing inconsistent render output across deploys) with a structured per-day form so the application controls the format, not the admin. The new model: 7 day entries (lun, mar, mie, jue, vie, sab, dom), each with `abierto: boolean` + `apertura: time | null` + `cierre: time | null`. The public `HoursSection` renders a single-line uniform string (e.g. "Lun a Vie 8:00 a 22:00 · Sáb 9:00 a 14:00 · Dom cerrado") generated by the app from the structure.

## Scope

### In Scope
- Drop `horario: String?` column, add `horarioJson: Json?` (nullable) to the Gym model
- Add `horarioDiaSchema` + `horarioSemanalSchema` Zod validators in `src/lib/schemas.ts`
- Swap the `horario` literal member of `gymFieldSchema` for `horarioJson` (value is `horarioSemanalSchema.nullable()`)
- Update `updateGymField` action, `GET /api/gym`, and `PATCH /api/gym` to use the new shape
- Build new `<WeeklyScheduleEditor>` client component with 7 day cards (each: switch Abierto/Cerrado + 2 `<input type="time">` pickers)
- Wire it into `GymConfigManager` Schedule sub-form (replace the textarea)
- Rewrite public `HoursSection` to format a single-line string from the structure
- Update unit tests (Zod schema) and E2E tests (admin per-day interactions + public render)
- Hard-drop existing `horario` data — no parse attempt (project is pre-launch, no real users)

### Out of Scope
- Cross-midnight schedules (e.g., 22:00 → 02:00 next day) — v1 limitation
- Per-day notes/holidays overrides — could be added later
- Multi-gym support (still singleton)
- Migrating the `unstable_cache` to `use cache` (separate follow-up)
- I18n for day names (Spanish labels are fine for v1)

## Capabilities

### New Capabilities
(none — extends existing capabilities only)

### Modified Capabilities
- `database`: Gym model — drop `horario`, add `horarioJson: Json?`
- `gym-config`: Admin form (Schedule sub-form), public render (HoursSection)
- `api`: GET /api/gym response shape, PATCH /api/gym validation
- `admin-panel`: Form layout for Schedule sub-form (7 day cards)

## Approach

Additive migration: drop old `horario` column + add new `horarioJson` column in a single migration. Backend (Zod, action, API, types) swapped in a coordinated commit. UI (WeeklyScheduleEditor + new HoursSection) in a follow-up commit. No dual-write deprecation period (project is pre-launch).

The `gymFieldSchema` discriminated union keeps the same `field` key strategy (one variant per editable field), but the `horario` variant is replaced by `horarioJson` carrying the entire weekly object. The form submits the whole object in one action call (not per-day saves).

Public `HoursSection` becomes a pure formatting function: given a `HorarioSemanal` object, produce a single-line string. Format rules:
- Consecutive open days with identical hours are grouped: "Lun a Vie 8:00 a 22:00"
- Each group separated by " · " (middle dot)
- Closed days: "Dom cerrado"
- If all 7 days closed: hide the entire section (null state)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| First `Json` column in the project — Prisma types it as `Prisma.JsonValue` (loose) | High | Narrow with Zod at every read boundary; treat as `unknown` until Zod-validated |
| `gymFieldSchema` Zod member swap is a breaking change for the existing schedule form | Medium | Coordinate form + schema + API in Slice 1; Slice 2 ships the new UI |
| Hard-drop of existing `horario` data loses any text already saved | Low | Project is pre-launch (no real users); admin re-saves in 30s; data format was free-text and incompatible anyway |
| Cross-midnight schedules not supported in v1 | Low | Document in spec; v2 can add `cierreSiguienteDia: boolean` |
| ~600-line delta exceeds 400-line budget | High | Chained PRs: Foundation (schema + Zod + API) + UI (form + render + tests) |

## Rollback Plan

Two-step rollback (worst case):
1. Revert Slice 2 (UI) → app falls back to no `horarioJson` in DB; admin form reverts to free-text `horario` field. (Requires re-adding the old field to the form, which is in git history.)
2. Revert Slice 1 (Foundation) → drop `horarioJson` column, re-add `horario` column. The data is gone but the form works again.

Since the change is in 2 chained PRs, rollback is also 2 steps in reverse order.

## Dependencies

- Existing `verifyAdmin()` for server actions
- Existing `useActionState` + `FormState` pattern
- Existing `FieldSubForm` + `useGymFieldForm` shell in `GymConfigManager` (Slice 2 of `gym-config-admin`) — reuse the pattern
- Existing time input pattern in `feriado-manager.tsx` (Slice 1 exploration noted this)
- Existing `unstable_cache` reader `getGymConfigForServer` (no changes needed — it just reads whatever fields exist)

## Performance & Breaking Changes

- **Bundle size**: Negligible — one new component (~150 lines)
- **Performance**: `horarioJson` is read together with the rest of the gym config; no extra DB roundtrips
- **Breaking changes**:
  - DB: column `horario` is DROPPED. Any data in it is lost. (Acceptable: pre-launch, free-text, incompatible format.)
  - API: `GET /api/gym` response shape changes (no more `horario` field; adds `horarioJson`).
  - Zod: `gymFieldSchema` `horario` variant is REPLACED by `horarioJson`. Any client importing the old variant breaks.
